# 성능 개선 기록

이 문서는 Issue 22 대응 과정에서 읽기 성능을 어떻게 단계적으로 정리했는지 기록한다.

측정 실행 규칙과 raw 리포트 해석은 [성능 측정 가이드](measurement-guide.md)를 기준으로 보고, 최종 비교 표와 결론은 [공식 재측정 결과](results/issue-22-remeasurement.md)를 기준으로 본다. 여기서는 각 단계에서 무엇을 왜 바꿨는지만 남긴다.

## 1차 개선. 조회 구조 단순화

### 배경

기준선에서 가장 무거운 구간은 채팅방 목록 조회였다. page size만큼 잘라서 응답하는 API였지만, 실제 구현은 그 전에 활성 채팅방 전체를 먼저 계산하는 흐름이었다. 이 때문에 응답 시간과 query count가 함께 증가했다.

메시지 히스토리 조회는 이미 최종 `cursorMessageId` 계약을 사용하고 있었지만, baseline 단계 구현은 계약을 맞추는 데 우선순위를 둔 단순 JPA/JPQL 조회였다. 따라서 1차 개선의 목적은 새로운 기능을 더하는 것이 아니라, 이미 고정된 읽기 계약을 더 단순한 조회 구조로 옮기는 것이었다.

### 변경 방향

이 단계에서는 캐시보다 조회 구조를 먼저 정리했다. 캐시는 숫자를 빠르게 줄일 수 있지만, 방 참여와 나가기, 읽음 처리, 마지막 메시지 반영처럼 무효화 규칙이 얽히는 순간 설명 비용이 커진다.

반면 QueryDSL projection과 cursor 기반 조회 정리는 API 계약을 유지한 채 읽기 비용을 줄일 수 있다. 이 프로젝트에서는 그쪽이 먼저 정리돼야 했다.

### 적용 내용

채팅방 목록 조회는 QueryDSL projection 중심으로 재구성했다. summary에 있는 마지막 메시지 정보와 정렬 기준을 그대로 사용하고, 응답에 필요한 값만 읽도록 바꿨다.

채팅방 목록 페이지네이션은 복합 cursor 기준으로 정리했다. `cursorLastMessageAt + cursorRoomId`로 다음 페이지를 계산하도록 맞췄다.

메시지 히스토리 조회는 `cursorMessageId` 기반 cursor pagination과 `(room_id, id desc)` 인덱스를 기준으로 정리했다. 이 단계에서는 API 계약을 바꾸지 않고, 기존 읽기 계약을 DB 접근 패턴과 맞추는 데 집중했다.

### 관찰

보존된 1차 재측정 요약 기준으로 채팅방 목록 조회는 same-machine 기준선 대비 median p95가 `1589.914ms -> 47.195ms`, query count가 `161 -> 1`로 줄었다.

1차 시점의 라운드별 raw/stability 스냅샷은 남아 있지 않고 대표 수치만 보존돼 있다. 그래도 방 목록 조회 개선의 기반이 이 단계에서 만들어졌다는 점은 분명하다.

메시지 히스토리 조회는 같은 방식으로 정리되지 않았다. 1차 요약 기준으로 latest 페이지는 `35.692ms -> 39.699ms`, cursor 페이지는 `86.883ms -> 120.251ms`로 모두 기준선보다 높은 수치가 나왔다.

## 2차 개선. 메시지 조회 읽기 경량화

### 배경

1차 개선 이후에도 메시지 히스토리 최신 페이지와 cursor 페이지는 모두 기준선보다 높은 수치가 나왔다. 따라서 메시지 조회 전체가 좋아졌다고 정리할 수 있는 상태는 아니었다.

읽기 API 경로도 여전히 무거운 편이었다. 멤버십 확인을 위해 엔티티를 읽고, 메시지 히스토리도 엔티티와 sender 연관 객체를 조회한 뒤 다시 응답 DTO로 옮기는 흐름이 남아 있었다.

참고로 cursor 검증 쿼리를 둘로 나누는 조정은 이 문서의 범위에 포함하지 않는다. 이 문서는 그 이후에 추가한 읽기 경량화만 다룬다.

### 변경 방향

이번 단계에서는 캐시보다 먼저 읽기 경로의 책임을 줄이는 쪽을 택했다. 메시지 조회 API가 실제로 필요한 정보는 많지 않다. 멤버십 확인은 ACTIVE 멤버 여부만 알면 되고, 메시지 히스토리 응답도 일부 필드만 사용한다.

따라서 필요한 것만 확인하고 필요한 값만 읽도록 바꾸는 편이 맞았다.

### 적용 내용

멤버십 확인은 `exists query`로 바꿨다. `existsByRoomIdAndUserIdAndStatus(...)`를 사용해 ACTIVE 멤버 여부만 확인하도록 조정했다.

메시지 히스토리는 `DTO projection`으로 조회하도록 바꿨다. 응답에 필요한 `messageId`, `senderId`, `senderNickname`, `content`, `type`, `createdAt`만 projection으로 읽게 조정했다.

쓰기 경로는 유지했다. 메시지 전송은 기존처럼 active member 엔티티와 ChatMessage 엔티티를 사용하고, 조회 경로만 별도로 가볍게 만들었다.

### 관찰

2차까지 반영한 최종 재측정 기준으로 메시지 히스토리 cursor 페이지는 1차 요약값 `120.251ms`에서 `76.373ms`로 내려갔고, same-machine 기준선 `86.883ms`보다도 낮아졌다. 반면 latest 페이지는 1차 요약값 `39.699ms`에서 `41.895ms`로 더 높게 나왔고, 기준선 `35.692ms`보다도 높았다.

즉 2차 개선에서는 cursor 페이지의 수치상 개선을 확인했고, latest 페이지는 읽기 경량화를 적용했지만 측정 편차로 인해 이번 결과만으로 수치적 개선 여부를 확정하지 않았다.
